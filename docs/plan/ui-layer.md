UI-Layer-Plan.md
Reference: Master-Implementation-Plan.md

Purpose
The UI Layer is a clean, robust, React+TypeScript application that:

Drives the user<>agent interaction experience
Surfaces all conversational, workflow, tool, and widget outputs in an intuitive, stream-aware, and context-rich manner
Adheres strictly to schema-driven rendering protocols—never open-ended LLM-generated UI code
Leverages Vercel AI SDK for best-in-class chat streaming, tool integration, and AI component foundations
Scope
Built on Next.js/React (App Router recommended)
Uses only typed, contract-derived props, not ad-hoc or "guessed" values
All UI flows (chat, widgets, interrupt states, approval events) map directly to explicit agent state and contract types
Code, components, event handling, and reactive patterns are all documented/auditable—never “magic”
Explicit Implementation Requirements
1. Component Structure & Conventions
🟦 AGENT:

Components are organized by function: ChatFeed, MessageInput, ToolOutputs, WidgetArea, Notification/InterruptBar, etc.
Use only TypeScript types derived from AgentState, Prompt outputs, and Tool outputs.
No component should ever render arbitrary HTML or React tree generated by LLM output.
Favor shadcn/ui and Vercel AI SDK primitives for chat input/feed, streaming, and tool displays.
🟨 HUMAN:

UI folder structure (e.g. components/, features/chat/, features/widgets/) must match domain concepts, not just technical patterns.
2. Streaming, Reactivity & Updates
🟦 AGENT:

Use Vercel AI SDK's streaming primitives (useChat, SSE/WebSocket client, etc.) to ensure the chat and widgets update in real time.
Partial responses or stepwise workflow updates must stream “in place” into the chat feed/UI, matching contract structures.
Intermediate agent events (tool begins, tool completes, needs approval, etc) should all trigger specific, tested UI states.
🟨 HUMAN:

Streaming skeleton/loaders and partial-update slots should be clearly marked (no “flash” or confusing state gaps during AI reasoning).
3. Widget Protocol Implementation
🟦 AGENT:

All widgets drawn in the UI must implement a schema-driven “widget protocol”:
Agent emits: { type: "chart", data: {...} } or similar—mapped to approved, prebuilt React component(s)
Never use .dangerouslySetInnerHTML or similar; all LLM/agent output to UI is sanitized and typed.
🟨 HUMAN:

Review every widget and the handler for contract drift or unsafe assumptions.
When in doubt, require an explicit agent and human review before adding new widget type.
4. Interrupt, Approval, Wait States
🟦 AGENT:

UI must always show clearly when agent or workflow is awaiting user input or human approval, with persistent actionable controls (approve, clarify, escalate, etc.).
Only task-specific prompts/interrupts are rendered—never freeform requests.
🟨 HUMAN:

UI UX for interrupts, waiting, or error states should be tested for usability, recoverability, and accessibility.
5. Error, Edge, and Fallback Handling
🟦 AGENT:

Build robust fallback and error UIs for:
Invalid agent/tool/widget output (schema reject)
Connection/interruption loss
Unexpected states (“out of workflow”, “prompt contract drift”)
🟨 HUMAN:

All error messages and edge-case flows should be copy reviewed and localized as necessary.
6. Theming, Accessibility, and Responsiveness
🟦 AGENT:

Use an accessible, mobile-ready, and optionally themable design system (ex: shadcn/ui + TailwindCSS).
All interactive elements should be keyboard navigable and screen-reader accessible.
Provide a dark mode (toggle, adaptive, or both).
🟨 HUMAN:

UX should be reviewed and validated using modern accessibility testing tools and user review.
7. Test Vectors
🟦 AGENT:

Storybook or equivalent UI test system with:
“Happy path” story for each chat/workflow/tool/widget state
Stories for interrupts, approval flows, errors, and ambiguous agent outputs
Manual and automated UI/E2E test (Playwright, Cypress, etc.) for all core workflows
8. Good / Bad Practice Callouts
Good:
Every UI state derives from agent/tool/widget schema—never “guessed”.
Partial outputs, streaming, and interrupts are evident and easy for the user to manage.
Widget protocol always schema-driven.
Error and wait states can always be explained.
Bad:
LLM output is rendered unsanitized, or as arbitrary UI code.
“Floating” UI states not tied to the contract.
No clear evidence for end-user about agent progress/corner cases.
Hidden or mysterious UX transitions.
